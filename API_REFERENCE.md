# NgpCraft Engine - API Reference

This document is the code-side reference for `NgpCraft Engine`.
It complements:
- `README.md` for user workflow and runtime integration

Scope:
- public Python API exposed by modules under `core/`, `i18n/`, `ui/` and `ui/tabs/`
- main data structures that are useful when extending the tool
- intended usage of each public function/class

Conventions:
- names prefixed with `_` are internal helpers
- `core/` is the main reusable API surface
- `ui/` and `ui/tabs/` expose Qt components used by the application; they are useful for internal extension, but less stable than `core/`

## Entry points

### `ngpcraft_engine.py`

Purpose:
- start the GUI application
- or run headless export / validation-suite generation from the CLI

Public API:

```python
main() -> int
```

Notes:
- default mode is GUI
- headless mode delegates to `core.headless_export.export_project()`
- `--validation-suite` delegates to `core.validation_suite.build_validation_suite()`
- `--validation-run` delegates to `core.validation_runner.run_validation_suite()`
- `--validation-run ... --build` also runs `make` in each generated project after export
- `--validation-run ... --build --smoke-run` also tries a runtime smoke launch when a ROM and emulator are available
- safe for build servers because `--export` skips `QApplication`

## Core modules

### `core/assets_autogen_mk.py`

Purpose:
- generate `assets_autogen.mk` from exported C assets

Public API:

```python
write_assets_autogen_mk(project_dir: Path, export_dir: Path) -> Path
```

Returns:
- path to the generated make fragment

**Link order is critical**: files are sorted via `_link_order_key` — sprites (`*_mspr.c`) first, maps (`*_map.c`) last, everything else in between.
Without this ordering, large map files can push sprites into ROM banks 0x21/0x22, making cc900 near pointers invalid → sprites invisible in-game.

### `core/template_integration.py`

Purpose:
- patch and align the generated exports with the NGPC template project
- optional autorun support is primarily meant for preview/validation; projects with an existing custom `src/main.c` may keep the generated assets/autogen files while disabling autorun integration

**RAM pool sizing (automatic)**: `patch_makefile_for_autogen` automatically injects the macros `NGPNG_AUTORUN_MAX_ENEMIES` / `MAX_FX` / `MAX_PROPS` / `MAX_REG` into the Makefile based on the actual entity counts in the project's scenes (`project_data`).
The default values (16/8/16/64) wasted ~980 bytes out of 8 KB total — the auto-calculation calibrates tightly with a safety margin (+2 enemies/props, +1 region).

Calculation: max across all scenes of (static enemies, largest single wave), +2 margin, minimum 4.
For waves: only the largest wave counts (not the total — avoids shmup overestimation).

To override manually, add AFTER the AUTOGEN section of the Makefile:
```makefile
CDEFS += -DNGPNG_AUTORUN_MAX_ENEMIES=8   # manual override
```

**Kill condition — platformer vs shmup**: the generated autorun code uses a shmup kill condition by default (`world_x < cam_px - 24`), which destroys enemies behind the camera. For a platformer with backtracking, replace with map bounds: `world_x < -24 || world_x > map_w*8+24`.
Applied in `template_integration.py` since 2026-03-15.

**Off-screen physics guard**: the autorun code includes a guard that skips expensive tilecol probes for enemies outside the viewport (>96px off-screen). Without this guard, every off-screen enemy still runs floor/wall probes, killing FPS when multiple enemies are present.
Applied in `template_integration.py` since 2026-03-15.

Public API:

```python
safe_ident(s: str) -> str
detect_template_root(*, project_dir: Path, project_data: dict) -> Path | None
default_export_dir_rel(project_data: dict) -> str
patch_makefile_for_autogen(
    *,
    template_root: Path,
    export_dir_rel: str,
    enable_autorun: bool = True,
    has_player_actors: bool = False,
    has_sound: bool = False,
    song_count: int = 0,
) -> tuple[bool, str]
write_autorun_main_c(

```

## Documentation maintenance

When a feature or public API changes, update these files together:
- `README.md`
  user-facing workflow, quickstart, generated files, runtime integration
- `API_REFERENCE.md`
  public module/class/function reference

Recommended update order:
1. update code and docstrings
2. update `API_REFERENCE.md`
3. update `README.md` if user workflow or integration changed

Practical rule:
- if a new public function/class is added in `core/`, it should appear in this file
- if a new tab/workflow is added, `README.md` should mention it
